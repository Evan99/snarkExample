Snark Hunting: Force Directed Graphs in D3

June 19, 2014

Authored by Steven Hall

Is it possible to write a blog post that combines d3.js, pseudo-classical JavaScript, graph theory, and Lewis Carroll?  Yes, THAT Lewis Carroll. The one who wrote Alice in Wonderland.  We are going to try it here.  

I thought I would mix in some fun historical trivia to keep it interesting as we check out force directed graphs in D3.  In this post we are going to develop a tool to load up, display, and manipulate multiple graphs for exploration using the pseudo-classical pattern in JavaScript.  We'll add in some useful features, a bit of style, and some cool animations to make a finished product (see the examples below).

Controls for examples

As usual, the demos presented here use a minimal amount of code.  There's only about 250 lines of JavaScript (if you exclude the comments) in these examples.  So it's enough to be a good template for your own project without requiring a ton of time to study and understand.  The code includes some useful lines to keep the visualization responsive (without requiring JQuery) and methods that do things like remove or add links or nodes.

There's also a fun "shake" method to help minimize tangles when the graph is displayed by agitating the nodes a little.  I find it annoying when the graph doesn't display correctly when it loads, so we'll take care of that.  Additionally, the examples incorporate a set of controls to help understand and explore the effect of the various D3 force layout parameters using the awesome dat.gui library from Google.  You can see a picture of the controls above.  We'll cover the controls in depth below, but first I'll introduce the examples and talk a little bit about the data.

Examples for this Blog Post

As always you can either look at the examples in your browser and "view source" to see what's happening in the code or you can download the full set of examples and supporting code from the GitHub repository.  The main classes are all found in the graph.js file which you take a look at here.

BONUS:  If you are working with adjacency lists from R, Python, etc you can compile your data from an Excel workbook.  Clone or download the GitHub repository then change into the directory.  Then:

// Compile adjacency lists from Excel workbook
npm install
node data/compile-data.js

// Fire up your viz on localhost
python -m SimpleHTTPServer 8080

Obviously, you need Node and NPM. The examples were created using this same script and you can just switch out the data in the Excel workbook and load up your data.  Worksheet names will be the names of the datasets you can choose in the visualization.  Check out the examples to get the idea.

About the Graph Data

Illustration for The Hunting of the Snark - Wikipedia

The graph data used in the examples is really a collection of smaller graphs that are referred to as snarks.  They are somewhat mysterious graphs with some interesting properties that you can read more about on the Wikipedia page.  They also can produce some pretty cool visualizations (not all of them, but a few are pretty fun to play around with).  I thought they would make a good foundation for writing about creating force directed graphs in D3.

Why Are They Called Snarks?

Alice in Wonderland is not only one of the most well-known children's stories, but is also a story filled with clever allusions to mathematical concepts and logical puzzles.  The pen name "Lewis Carroll" was used by a complicated man (accused of being a pedophile...among other thing) whose real name was Charles Dodgson.  In addition to being a master story teller, he happened to be a mathematician and logician who also wrote numerous texts on a variety of mathematical subjects.  He worked most of his career as a mathematician at Christ Church College, Oxford.  If you're interested in some of the mathematical ideas embedded in Alice in Wonderland you may want to check out this part of the Wikipedia page or this article.

Example Flower Snarks - Wikipedia

Carroll also wrote a lesser known nonsense poem called The Hunting of the Snark.  In the poem the snark is a mysterious class of creature that is never fully described and at the end of the story the reader is left wondering what the creature even looks like.  Carrol himself would never elaborate on what exactly a snark was, preferring to say "he did not know and could not explain."

And so snark seemed like a perfect name for a family of graphs that remains mysterious and difficult to describe.  From the Wikipedia page on snarks:

There are also really fun to look at and make a good launching point for talking about D3's force layout.  After some searching, I found a dataset that contains the node and link information for many common snark graphs (thank you, internet!).  You can get the original data here.

Looking at the Code

Creating a D3 viz is less commonly done using a pseudo-classical style, but can be a powerful way of organizing your code if you need to develop visualizations that are more like small applications or tools that provide complex interactions rather than just a simple chart or graph.  When I first started I was constantly running into walls when I needed to add a new feature or provide some interactivity to my data visualizations.  Learning more about the various JavaScript patterns can give you a nice tool box to draw from in your work.

If you are unfamiliar with the pseudo-classical pattern, you can take a look at this post I did a few weeks back that gives you a quick overview of the most common patterns.  We are going to develop a graph class that can dynamically load, add/remove nodes, and allow you to quickly adjust the various parameters on the d3.force.layout (like charge and gravity) to create the view you want to see with your graph data.

A graph at its most fundamental is a set of vertices and edges.  In the D3 world, vertices are usually referred to as "nodes" and edges are referred to as "links."  In D3, in order to get a graph to the screen, we need to deliver our nodes and links as arrays of objects to the force layout. You can send an array of arrays that contain numbers (aka adjacency lists), but D3 behind the scenes converts these to objects because it needs to add additional properties in order to make the simulations run.  We'll look at that process more closely below, but let's take a look at what's happening in the main JavaScript file in the examples first.

In these examples, the bulk of the action is happening in the graph.js file, so it will make sense to focus in on that file.  If you look in that file, you'll see classes for all the things we will need:  Config, Node, Link and Graph.  The only objects we need to interact with directly are Config and Graph.  You'll find in each HTML file for the examples that the process is exactly the same: (1) load our data, (2) create a Config instance and (3) make a new Graph by passing it the data and configuration information.  The rest of the graph.js file, for the most part, is defining a set of methods on the graph.prototype that will give us all the functionality that we need like adding and removing node or links.

The code in full for each of the example HTML files looks like this:

// CODE FROM THE HTML TO INITIATE THE PROCESS
d3.json('data/snarks.json', function (error, data) {  //A

  var config = new Config({  //B
    height: window.innerHeight,
    width: window.innerWidth,
    dataset: 'snark56',
    gravity: 0.21,
    charge: -550,
    theta: 0.9,
    friction: 0.9,
    distance: 10,
    strength: 0.9
  });

  window.graph = new Graph(data, config); //C
  graph.update(); //C
  graph.shake();  //D
  window.addEventListener('resize', graph.onResize.bind(graph), true); //E
});

As usual when doing D3 data visualizations, we begin at point A with an asynchronous call to fetch the data file that has the node and link information for all the snark graphs.  With the data loaded a Config object is created with all our initializing values.  This Config instance will become important for set up our controls in dat.gui below. 

The instance of graph created on page load

At point C above we create a new graph and make it a global (i.e. window.graph), simply so it can be inspected in the console for these examples.  You would not normally do this because, as we all know, global variables are bad news. But in this case you can interact with the graph in the console: adding and removing nodes etc (call graph.update() after to make your changes reflect on the screen).  The new graph gets passed the data file and all the configuration information.  What we get back is an object like the one seen in the screen shot to the left.

The object that gets created is more than simply the graph itself. The instance of Graph has all the properties and methods needed to switch out datasets, add or remove nodes, update the screen, and also contains an instance of dat.gui (the controls).  You can see all the methods that the graph object has by inspecting the prototype object in the console (this is __proto__ in chrome).

 

 

D3 Force Layouts

When you run graph.force.start() in these code examples you are running a physics simulation in the browser.  D3 gives you a set of parameters that you can use to control how the objects behave which we'll look at below in the section on dat.gui.  Here, it's important to point out that, when working with the D3 force layout (just like some other layouts) is that it is modifying your node and link objects when the simulation is running.  Let's take the nodes as an example.  If you look in the graph.js file, the Node class is dead simple:

var Node = function(obj) {
  this.id = obj.id;
};

Node with new attributes added by D3

That's it.  We just give each of our nodes a unique id.  We could add other info we wanted to store there as well, but that's the minimum we need to make this simple set of examples work. But if you look at an actual node in the console, you can see that D3 is augmenting our object with its own properties (see the screenshot to the left) in order to control its position during the simulation.  Properties for weight, index, px, py, x, and y are needed for D3 determine where the nodes (and links as well) are and where they're going on the next tick of the simulation.  It's important not store information with these property names on your object because they will get over written.

Once we have given the force layout our arrays of nodes and links, there are some other parameters that can be set on the force layout to make it behave in different ways.   I think the best way to understand those parameters, like charge, friction, theta, etc, is to be able to play with them in the browser on a real graph.  In the next section we'll wire up the controls for doing this.

Hooking Up the DAT.GUI Controls

Dat.gui is one of those great ideas that the wizards at Google came up with.  It's a super lightweight way to make a simple interface for your visualization.  Even better for me, the documentation is really great so you can take a look at that in order to get all the details.  We only need to say a few things here.  In making these examples I tried to pull out all the key knobs you want to be able to play with on a D3 force layout and put them in a single object that can be manipulated by dat.gui.  There are some other that could be added but I think all the key parameters are there (if you want to add more it is pretty straightforward).

The Config class in the graph.js file will take whatever properties you pass in on an object and create an new instance with those properties.  The class looks like this:

var Config = function (options) {
  for (var opt in options) {
    this[opt] = options[opt];
  }
};

In the HTML file for the examples we create an instance and pass that into our graph along with the data.  It looks like this:

var config = new Config({
  height: window.innerHeight,
  width: window.innerWidth,
  dataset: 'snark40',
  gravity: 0.21,
  charge: -550,
  theta: 0.9,
  friction: 0.9,
  distance: 10,
  strength: 0.9
});

window.graph = new Graph(data, config);

Inside of the Graph class we do the rest of the wiring when a new Graph instance is created.  For each variable that we want to appear in our controls we need to pass in a function that tells dat.gui what to do when the value changes.  For example, the code for the gravity parameter looks like this:

// Inside the Graph class in graph.js
this.datgui = new dat.GUI();

this.datgui.add(this.config, 'gravity', -0.1, 1).onChange(function (d) {
  this.force.gravity(d);
  this.force.start()
}.bind(this));

This tells dat.gui that we want a control for a property called 'gravity' on the this.config object.  The range of input values for the control should be -0.1 to 1.  We then pass in a function that changes the gravity parameter on the force layout and then starts (restarts) the simulation.  We also bind this function so it will always run in the context of the Graph instance. 

The code to set up the controls using dat.gui ended up being pretty verbose, but the payoff (I think) is worth it.  Playing with the controls really helps understand how each of the different parameters will impact the graph.  Very cool.
